# CPU为了提高指令并行效率，实现了超线程技术，和OS的线程没有关系

# GMP模型
- 协程调度过程的三个重要角色
  - G
    - Goroutine
        - Go协程
        - 存储了协程的执行栈信息、状态和任务函数等。初始栈大小约为2~4k，理论上开启百万个Goroutine不是问题
  - M
    - Machine Thread
        - 对系统线程抽象、封装
        - 所有代码最终都要在系统线程上运行，协程也是代码，也不例外
  - P
    - Processor
        - 虚拟处理器（不是CPU）
        - - 可以通过环境变量GOMAXGROCS或runtime.GOMAXPROCS()设置，默认为CPU核心数，建议不要超过CPU核心数
        - - P的数量决定着最大可并行的G的数量
        - - P有自己的队列（也称为本地队列LRQ，长度256），里面放着待执行的G
        - - M和P需要绑定在一起，这样P队列中的G才能真正在线程上执行

- 协程调度过程
  - 1、使用go func创建一个goroutine g1
  - 2、当前P为p1，将g1加入当前P的本地队列LRQ（Local Run Queue）。如果LRQ满了，就加入到全局队列GRQ
  - 3、p1和m1绑定，定期会扫描全局队列GRQ拿一个，m1先尝试从p1的本地队列LRQ中请求G
    - 如果没有拿到，就从全局队列GRQ请求G
    - 如果还没有拿到，就从network poller中拿一个runnable的G
    - 如果还没有拿到，就随机从别的P的本地队列LRQ中偷（work stealing机制）一半G到本地队列LRQ中，这是一种负载均衡
  - 4、假设m1最终拿到了g1
  - 5、执行，让g1的代码m1线程上运行，有如下几种情况
    - - g1正常执行完了（函数调用完成了），g1和m1解绑，执行第3步的获取下一个可执行的g
    - - g1中代码主动让出控制权，g1和m1解绑，将g1加入到全局队列GRQ中，执行第3步的获取下一个可执行的g
    - - g1中进行channel、互斥锁等其他操作进入阻塞态，g1和m1解绑，执行第3步的获取下一个可执行的g。如果阻塞态的g1被其他协程唤醒后，
      - 就尝试加入到唤醒者的本地队列LRQ中，如果本地队列LRQ满了，就连同g的本地队列LRQ中一半转移到全局队列GRQ中
    - - 操作系统
        - ① 同步系统调用时（分手hand off机制）
            - 如果遇到了同步阻塞系统调用，g1阻塞，m1也被阻塞了，m1和g1解绑
            - 从休眠线程队列中获取一个空闲线程，和p1绑定，并从p1队列中获取下一个可执行的g来执行；如果休眠队列中无空闲线程，就创建
          - 一个线程提供给p1
            - 如果m1阻塞结束，g1优先回到原来的p1的本地队列LRQ。如果不行，加入到一个空闲的p的本地队列LRQ。如果没有空闲的p，g1就会
          - 放到全局队列GRQ中。m1加入到休眠线程队列中
        - ② 异步网络IO调用时
            - 网络IO代码会被Go在底层变成非阻塞IO，这样就可以使用IO多路复用了
            - m1执行g1，执行过程发生了非阻塞IO调用（读/写）时，g1和m1解绑，g1会被网络轮询器Netpoller接手。m1再从p1的本地队列
          - LRQ中获取下一个Goroutine g2执行。注意，m1和p1不解绑
            - g1等待的IO就绪后，g1从网络轮询器移回p的本地队列LRQ或全局队列GRQ，重新进入状态
