# 接口interface
- 用来把不同类型的值，用方法来发现关系。接口里面放着方法的声明，接口更加关注操作，而方法又是函数，函数就是操作。
- 约束、规范，类似与Java中的接口，是一组行为规范的集合，为了减少耦合
  - 类比谈判，遵守规范才能谈，否则免谈
- Go语言中，为了实现面向对象的类似的功能，用到了结构体的父子结构和接口技术
- 接口非常重要，项目里会经常使用。应减少耦合，以低的代价实现关联
- 规范
  - 只有操作（函数、方法）的声明，注意：没有声明（只有声明，不实现）
  - 谁遵守该接口规则，谁就实现
  - 不同的类型实现接口，方法函数体内部的具体实现可以完全不一样
  - 语法

```go
            type 接口名 interface {
```

      - 方法1 (参数列表1) 返回值列表1
      - 方法2 (参数列表2) 返回值列表2
      - ...

```txt
            }
```

    - 规则
      - - 接口命名习惯在接口名后面加上er后缀
      - - 参数列表、返回值列表参数名可以不写
      - - 如果要在包外使用接口，接口名应该首字母大写；方法要在包外使用，方法名也要首字母大写
      - - 接口中的方法应该设计合理，不要太多
        - 方法
          - - 方法名称
          - - 入参列表
          - - 出参列表
          - 所有方法必须全部实现（一个都不能少），且方法名、入参列表、出参列表完全一致
          - 至于程序员对方法内部如何实现函数体，接口并不关心（接口只关心方法名、入参列表、出参列表）
          - Go语言中，接口建议定义为小接口，即接口中方法尽量少，通过小接口组成大接口
- 与Java的区别
  - Java

```txt
            class A implements 接口1, 接口2, 接口3
```

      - 一定要实现接口1的所有方法
      - 一定要实现接口2的所有方法
      - 由于是明着写明了实现哪些接口，编译器会检查
  - Go
    - 代码中不需要明着写implements什么接口，只要一个结构体或其他方法类型实现了接口所有的方法，编译器就认为实现了该接口
    - 如果对接口1中的某个方法没有实现，编译器也不会报错，编译会认为未实现该接口

- 实例的指针
  - 接口变量为实例的指针
    - 实例receiver，可以调用（注意调用的是实例的副本）
    - 实例的指针receiver，可以调用。推荐
  - 接口变量为实例
    - 实例receiver，可以调用（注意调用的是实例的副本）
    - 实例的指针receiver，不能调用

- 空接口

```go
        var x interface
```

    - x变量，变量的类型interface{}
    - interface{}接口类型
      - 该接口没有任何要实现的方法
      - 由于interface{}接口类型没有方法，那么就可以说任何类型都实现了该接口
      - 任何一个非接口类型的值，都有自己的类型，就可以说这些值既是他们自己的类型，也是interface{}接口类型的
  - 空接口类型变量，可以接收任何合法类型的值
  - type any = interface{}

# 接口的应用
- 1、返回值
  - 最后一个参数可以是error接口类型的
    - nil 表示没有问题，没有nil说明有错误
- 2、抛出一个错误
  - panic(errors.New("错误原因或描述")
  - x := recover()  // any，对x类型断言
- 3、多态
  - 设计很多结构体，用来装业务数据，操作的方法
    - 有可能有父子关系
    - 为了灵活应用多态，Go采用了接口
- 4、接收任意类型，少用 []any
- 5、不同模块之间设计接口（方法的声明）
  - 模块的类型上具体方法接口方法
