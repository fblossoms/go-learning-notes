# Channel通道
- Channel底层是一个先进先出的环形队列
    - full或empty就会阻塞
    - send发送
    - recv接收并移除
    - sendx表示最后一次插入元素的index
    - recvx表示最后一次接收元素的index
    - 发送、接收的操作符都是<-

- 构建
    - nil channel不可用，不要用，没法用
    - 无缓冲通道unbuffered channel（同步通道），make(chan int, 0)，容量cap为0，长度len为零，0个元素就满了
      - 先send发送就阻塞，直到有recv接收进来
      - 先recv接收就阻塞，直到有send发送进来
    - 缓冲通道buffered channel

```go
            make(chan int, n) n >= 1
```

      - make(chan int, 1) 只能容纳一个元素
        - 先send一个元素，不卡，再send就要阻塞，直到有别人（协程）recv
        - 先recv一个元素，卡，直到有别人（协程）send进来
    - 构造时，使用make
- 单向通道
    - 只读 <-chan type
    - 只写 chan<- type
    - 定义只读或只写的通道没有使用的意义
    - 用在形参定义中，用来约束在函数中，通道的操作，避免误操作

- 通道的关闭
    - 只能关闭一次
      - 关闭后不能再send，不是清除，当没有引用时，会被标记为垃圾回收
      - 可以for遍历完里面的容量，没有存量for不不再遍历，for退出了
      - 手动<-ch可以再读取，读完存量后，可以再读，如何判断数据有效呢？返回第二值查看
        - 如果第二值是true，说明是存量
        - 如果第二值是false，说明读取的不是存量，返回的是元素的零值
    - 不是关闭的
      - 无穷的容器，读不完，没有元素就阻塞，不管是不是读取一次还是for迭代

- 通道的遍历
    - nil通道，别想了，没法用
    - 有缓冲通道
      - 未关闭
        - 无穷个元素，单个读取、for遍历都不可能读完，当队列为空，就阻塞
      - 已关闭
        - 有穷个元素，单个读取、for遍历一定可以读完，for自己识别是否还有存量，单个读取要判断第二值
    - 无缓冲通道
      - 未关闭
        - 无穷个元素，单个读取、for遍历都不可能读完，当前队列为空，就阻塞
      - 已关闭
        - 有穷个元素，单个读取、for遍历一定可以读完，for自己识别是否还有存量，单个读取要判断第二值

- 通道死锁
    - channel满了，就阻塞写；channel空了就阻塞读。容量为0的通道可以理解为0个元素满了
    - 阻塞了当前协程之后就会交出CPU，去执行其他协程，希望其他协程帮助自己解除阻塞
    - main函数结束了，整个进程就结束了
    - 如果在main协程中，执行语句阻塞时，环顾四周，如果已经没有其他协程可以执行，只剩主协程自己，解锁无望了，就报一个fatal error deadlock
    - 如果通道阻塞不在main协程中发生，而是发生在子协程中，子协程就会继续阻塞着。但是由于至少main协程是一个值得等待的希望，运行时不能帮助识别到死锁。如果真的
  - 没有任何协程帮助这些子协程接触阻塞状态，那么事实上这些子协程解锁无望，已经基本是死锁了，虽然这时候主协程还活着
    - 死锁的危害可能会导致进程活着，实际上某些协程未能真正工作而阻塞了，应该有良好的编码能力和习惯，来减少死锁的出现
