# 哈希表

- 加密和解密
  - 加密：把一段信息（明文，如字符串abc），将byte切片转换成另一种样子的byte切片，即明文 --加密算法-> 密文
    - 对称加密：加密和解密用的同一个密钥
    - 非对称加密：加密和解密用的不是同一个密钥或算法
  - 解密：密文 --解密算法-> 明文

- 哈希算法（称为hash，哈希，单向散列）：单向加密算法
  - 把任意长度的输入值通过算法变换成固定长度的输出值
  - 例如[]byte --哈希算法-> [128]byte，本质上可以理解为大整数

  - 特征
    - 单向：没有或济南找到逆运算
    - 给定一个x，一定得到一个固定的y（y = hash(x)）
    - 冲突：由于输出空间有限，而输入空间无限，可以找到两个输入值x1和x2，通过hash算法得到同样的y
    - 效率高
    - 算法设计上，x值微小的变换（哪怕是1bit），都会引起y巨大的变化

  - 常见算法
    - Md5信息摘要算法
      - 输出128bit（16byte），一个字节变成两个字符输出（每1个字节=2个十六进制字符），最终呈现32byte的字符串
      - 密码明文加密，不要在数据库中存储明文的密码
      - 快速比较文件
        - 比较路径是否一致
        - 文件字节数不用比（无效比较）
        - 把文件看作[]byte --Md5-> 字节序列
      - 下载、上传时
      - 数据库中字段值比较

    - SHA-xxx安全散列算法，xxx可取1、224、256、384、512，常用256+
      - 使用数字签名（类似指纹，具有独特性），SHA-512签名的结果是一个[64]byte、string-128
        - 签名的原文明文是什么？在token（base64）里面有payload原文（header+payload+signature）
        - 在通过SHA算法（必须使用密码secret），可以得到一个签名signature，签名无法被修改
        - Header+payload原文修改后的 Header`+payload` + signature
          - 验证：hash(Header`+payload`)将得到signature`，即此时签名发生了变化，服务器验证不通过
        - token修改了h`+payload`，通过密码secret密码`得到的signature`
          - 验证：signature`与signature相比较，此时签名发生了变化，服务器验证不通过

    - 算法理论上不可逆，暴力穷举是可以破解密码的
      - Md5算法太快，可以通过映射表（彩虹表）反推

- map映射，字典（Python的dict）
  - 底层原理、扩容策略面试题

  - nil map（零值map）不可用
  - key键 -> value值
    - 关心的是数据，核心数据就是value
    - key为了区分（检索），方便定位
  - 底层基于hash table哈希表
    - 存的是 键值对（常见英文名称）
      - key-value pairs
      - item kv
      - entry
    - 容器
      - 长度可变，即键值对的个数可变，长度依然是元素的个数
      - 容量cap()存在，但是不能使用（不给探测哈希表容量），能用len
      - 键值对在容器中按照key的hash值散放
    - 没有索引index，线性表有索引[0, len-1]
    - 有key，用来定位value
      - hash算法
    - key无需：键值对没有先后顺序，存储依赖key
    - key去重，重复键会覆盖旧的值，同一个键对应的值只能留一个（有且只有一个键）

    - bucket桶
      - 用来放键值对
      - 每个桶里面可以有任意个键值对
      - bucketIndex := hash(key) mod 2^B

  - map是引用类型
    - hmap（Header Map）结构体
      - len，属性名叫做count，即当前键值对的个数
      - B，uint8，2^B（2的B次幂），代表有多少个bucket桶个数，桶构成的数组的长度
      - buckets
        - 指针，找桶数组在哪

  - 增删改查CRUD
    - 增C
      - count +1
      - 算出bucketIndex，如果key不存在，就在桶header对应的链表增加键值对
    - 改U
      - 算出bucketIndex，如果key存在，就覆盖原键值对的值
    - 删D
      - count -1
      - 必须给key，根据key找到bucketIndex。如果找到了key旧删除键值对，如果没找到，就报错
    - 查R
      - 按照key定位
        - 对于哈希表来说，都要用key，因为效率是最高的，时间复杂度O(1)
        - 遍历链表找key
          - 链表可以很长，但是链表长效率不高，可以加bucket（桶数组扩容），很少缩容
            - 若LoadFactor（负载因子）：count / 2^B > 6.5，说明链表平均过长，考虑扩容操作
          - 极端情况，给的key经过hash算法取模值都偏了，bucketIndex索引一致，键值对集中在某个bucket链表
      - 按照value（没有意义）
        - 按照value需要先找到所有键值对，才能比较value，所以没有意义

    - 遍历
      - 只要是容器，就一定那个可以遍历
      - 把容器里的所有元素不重复的读取一遍
      - 对于非线性数据结构，想要转成线性数据结构，可以先遍历元素，把遍历到的元素一个一个装到切片（线性数据结构）中
      - 遍历效率与数据结构无关，只和规模有关

  - 扩容
    - 当LoadFactor（负载因子）：count / 2^B > 6.5启用扩容策略
    - 增量扩容
      - B++，相当于 *2，即桶数组翻倍
      - 渐进式迁移方案，每次迁移只需要移动详单与上一行的一般
      - 新桶：新数据进新桶数组的对应索引的桶里
      - 写入操作：顺便把老童数组里面至多两个桶做迁移
    - 等量扩容（Go特有）
      - 桶里指向底层拉链法的链表，链表的每一个节点node是一个8个元素的数组
      - 等量，桶数组长度不变，键值对个数不变
      - 由于Go链表节点的特殊性，遇到多次删除后，有些元素删除了，但是节点形成数据空洞，为了回收数据空洞，规定达到一定阈值后，触发等量扩容
      - 规整链表的节点

  - Go中的map并不是并发安全的数据结构，如果出现并发读同时有并发写，或并发写同时有并发写，会抛出异常
