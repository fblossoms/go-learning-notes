# 数据结构和算法：解决问题的方法

- 指令和数据（从当前运行的进程角度出发看内存）
  - 数据放到内存（Memory）中，内存是分割成了小房间（每个房间 1 Byte），为每个房间编号，理论上从0开始编号，编号必须连续（门牌号相当于指针）
  - 如果放数据时一个字节不够用，就连续占用多个字节
  - 数据类型 在 内存中 是有结构的
    - 立体结构：数据在内存中的立体结构不同，决定了使用方式不同，这将影响效率
      - 线性表
      - 哈希表
      - 树（Tree）
      - 图（map）
    - 线性表和哈希表占用日常使用的80%以上
  - 数据结构是利用内存的方式，与使用的语言无关，所有语言都必须实现对数据类型的支持
  - 一定要为某类型的数据结构找到好的处理方法（好的算法）

# 线性表（简称表）

- 逻辑概念/数学概念：一组数据（0个或任意个），多个元素的集合（不是去重集合collection），元素必须确定先后顺序，排成一队
  - 一个元素的前面的数据称为前驱元素，后面的称为后继元素
  - 有序序列（sequence）：元素有前后顺序。
  - 不是排过序的
  - （sorted），排过序的（ordered）

- 物理实现
  - 顺序表
    - 内存中找一段连续的内存空间，把元素依次放在其中
    - 每个元素的前后顺序，可以用内存的线性增加表达（地址增加方向）
    - 每个元素占用一样的字节数，每个元素有字节的类型，如果语言允许元素的类型不一样，他也会调整成一样
    - 缺点：开辟连续空间，不够灵活，有危险性
  - 链接表（简称链表）
    - 内存中元素并不一定挨在一起存放，每一个元素在内存中找到合适的位置存放即可（每个元素几乎不会使用连续的内存空间，即大概率随机分布）
    - 通过header连接首个元素，各元素顺序通过指针指向前后元素建立联系
    - 缺点：内存利用效率相对低，查询效率相对低

# 顺序表：例如数组（不可以增删元素）、切片（底层依赖顺序表（数组））

- 可索引：可以依次编号（称为索引或下标），一般从0开始，最大值是长度-1

- 定位元素：首地址 + 索引（index）* 每个元素占用的字节，首地址 + 偏移offset

- 底层有元数据（meta），带length信息，当使用len时就会返回meta中的length，从而指导顺序表的长度，确定索引值的范围

- 增删改查（CRUD）：都存在定位问题
  - 增（C create 增加元素）：长度len + 1
    - 首部（insert(0, value)）：在最后开辟新空间（字节占用），现有元素全部向后复制并覆盖，把增加元素的值覆盖在首部
    - 中间（insert(n, value)）：插队，在最后开辟新空间，插入点之后的数据依次（从后往前）向后复制并覆盖（拷贝），把增加元素的值覆盖到插入点
    - 结尾（push / append，代价最小）：在最后开辟新空间（字节占用），然后把增加元素的值覆盖在该空间
    - 扩容原则：增加元素，就要增加占用空间，在连续空间的后面增加
      - 如果后面有空间，直接占用
      - 如果后面没空间，新找一片连续的空间
        - 如果没有连续空间，内存空洞多，碎片化内存
          - 规整、平整出一大块连续空间（垃圾回收）
          - stw（stop the world暂停代码指令）应尽量短和少，程序速度变慢可能是在进行垃圾回收

  - 删（D delete 删除元素）：长度len - 1
    - 首部（remove(0, value)）：把删除元素的值删除，现有元素全部向前复制并覆盖，然后回收最后的空间，表现为该空间不可访问不可用
    - 中间（remove(n, value)）：把删除元素的值删除，在最后开辟新空间现有元素全部向前复制并覆盖，然后回收最后的空间（字节占用）
    - 结尾（pop / remove，代价最小）：把删除元素的值删除，然后回收最后的空间（字节占用）
  - 数据规模大时， 顺序表不适合对首部、中间进行增删操作
  - 改（U update 更改元素）：长度不变
    - 主要在定位，找到后更新内容
  - 查（R retrieve 查找元素）：长度不变
    - 定位找内容
      - 索引定位：首地址 + 索引（index）* 每个元素占用的字节，首地址 + 偏移offset。分两步（O(2)与规模n无关，常数级都记作O(1)）
      - 值定位（效率低）：根据偏移，从前向后找。幸运的话第一个就找到了（O(1)），不幸晕的话遍历完都没找到（O(n)）

- 遍历
  - for、for-range（遍历所有元素）
  - 容器中所有的元素不重复（不同索引上）的拿一遍
  - 按照索引从0开始向后拿 长度-1的索引

- 复杂度
  - 时间复杂度（优先考虑）：O(规模) n表示元素个数，n越大表示规模越大
  - 空间复杂度：算法消耗的内存

# 链接表（使用标准库中的container / list实现双向链表）

- 元素element（也称为节点node）
  - 每个元素包含值（value）和指针（next（指向后一个元素）或prev（指向前一个元素））
  - next指针指向下一个元素的首地址，最后一个元素的next指针为零值（悬空）
- 单向链表（next指针）：从头（header）开始使用next找到结尾（tailer）
- 双向链表（next指针和prev指针）：从头开始使用next指针找到结尾，也可以从结尾开始使用prev指针找到开头
- 长度：增加元素 加1，减少元素 减1，元数据（mate）中储存
- 可以索引，分为正索引（[0, len-1]）和负索引（[len-1开始为-1，逐步减1]）

- 增删改查（CRUD）：都存在定位问题
  - 增（C create 增加元素）：长度len + 1
    - 首部：header指向新头部，新头部的next指向原头部，原头部的prev指向新头部
    - 中间：header、tailer不变，插入元素n的prev指针指向前一个元素，next指针指向下一个元素。插入点的原前一个元素（next）和原后一个元素（prev）指针断开连接，然后分别指向插入元素
    - 结尾（append）：tailer指向新尾部（覆盖整数），原尾部的next指向新的尾部，新尾部的prev指向原尾部
    - 扩容原则：不需要扩容，只需要构建新元素的内存空间，大概占24字节（value值8，next指针8，prev指针8）

  - 删（D delete 删除元素）：长度len - 1
    - 首部：header指向被删除元素的下一元素，下一元素（prev）与被删除元素（next）断开链接
    - 中间：header、tailer不变，被删除元素的前一元素（next）和后一元素（prev）建立新链接，他们同时与被删除元素断开链接
    - 结尾：tailer指向被删除元素的上一元素，上一元素（next）与被删除元素（prev）断开链接
  - 数据规模大时，链接表适合对首部、中间进行增删操作
  - 改（U update 更改元素）：长度不变
    - 主要在定位，找到后更新内容
  - 查（R retrieve 查找元素）：长度不变
    - 定位找内容
      - 索引定位
        - 头尾：有header和tailer指针，索引0、索引len-1，时间复杂度O(1)
        - 中间：遍历到索引位置，由于使用指针来找元素，效率较低（相对于顺序表），可以分多个遍历变量，例如两步跳、三步跳等
      - 值定位（效率低）：从前向后找。幸运的话第一个就找到了（O(1)），不幸晕的话遍历完都没找到（O(n)）
      - 顺序表和链接表都建议使用索引查找

- 思考题：
  - 想知道线性表中有没有某个内容，请问这个线性表采用什么物理存储方式合适？
    - 都不合适，根据内容查找效率低
  - 使用索引插入数据或删除数据，哪种物理存储方式合适？
    - 数据量大时，采用链接表；数据量小时，顺序表和链接表均可
  - 在线性表中随意增删数据，且使用索引定位数据位置，用哪种物理存储方式合适？
    - 数据量大时，采用链接表；数据量小时，顺序表和链接表均可
  - 线性数据结构，如果增删较多，且发生在头尾？哪种物理存储方式合适？
    - 数据量大时，采用链接表；数据量小时，顺序表和链接表均可

- 总结：
  - 使用链接表场景居多
  - 时间复杂度和空间复杂度优先考虑时间复杂度最优
  - 会分析时间、空间利用并找到最好的数据结构和算法是称为资深工程师的必备技能
